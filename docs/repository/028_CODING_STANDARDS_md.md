## PURPOSE  
This document defines the official coding standards for the project.  
Its objective is to ensure that every line of code generated by Claude remains:  
* Consistent  
* Readable  
* Maintainable  
* Predictable  
* Testable  
* Production Ready  
Code quality is considered an architectural requirement.  
   
⸻  
   
## GENERAL PRINCIPLES  
Claude shall always write code that is:  
* Simple  
* Explicit  
* Typed  
* Modular  
* Reusable  
* Deterministic  
* Documented  
Complex code requires architectural justification.  
   
⸻  
   
## LANGUAGE  
Primary Language  
TypeScript  
Configuration  
Strict Mode  
Mandatory.  
The use of any is prohibited except when explicitly justified and documented.  
   
⸻  
   
## FILE SIZE  
Recommended limits:  
React Components  
≤ 300 lines  
Business Services  
≤ 500 lines  
Utility Files  
≤ 200 lines  
Configuration Files  
As required  
Large files should be split by responsibility.  
   
⸻  
   
## SINGLE RESPONSIBILITY  
Every file shall have one responsibility.  
Examples:  
Financial Service  
↓  
Financial Logic Only  
Formula Registry  
↓  
Formula Definitions Only  
Chart Component  
↓  
Visualization Only  
Mixed responsibilities are prohibited.  
   
⸻  
   
## NAMING CONVENTIONS  
Use descriptive names.  
Good Examples:  
```
calculateNetProfit()

createInventoryLayer()

syncOrders()

validateSettlement()

generateDailySnapshot()

```
Avoid abbreviations.  
Business terminology should match Repository documentation.  
   
⸻  
   
## VARIABLE NAMING  
Variables should describe business meaning.  
Examples:  
```
grossProfit

shippingSubsidy

remainingInventory

formulaVersion

expectedSettlement

inventoryLayer

```
Avoid generic names such as:  
```
data

value

item

temp

obj

```
unless context is extremely limited.  
   
⸻  
   
## FUNCTION DESIGN  
Functions should:  
* Perform one task.  
* Have clear inputs.  
* Return predictable outputs.  
* Avoid side effects where possible.  
Preferred structure:  
Validate  
↓  
Execute  
↓  
Return  
   
⸻  
   
## FUNCTION LENGTH  
Recommended maximum:  
50 lines  
Longer functions should be decomposed into smaller reusable units.  
   
⸻  
   
## CLASS USAGE  
Prefer:  
Pure Functions  
↓  
Services  
↓  
Classes only when justified.  
Classes are appropriate for:  
* Adapters  
* Repositories  
* Complex Domain Objects  
Avoid unnecessary object-oriented design.  
   
⸻  
   
## COMMENTS  
Comments explain:  
Why  
Not  
What  
Bad Example:  
```
// Increment i

```
Good Example:  
```
// Shipping subsidy is calculated separately because customer-paid shipping
// is treated independently from actual shipping expense.

```
Code should remain self-explanatory whenever possible.

## TYPES  
Every public interface shall define explicit types.  
Example:  
```
interface FinancialAdjustment {
  id: string;
  orderId: string;
  type: AdjustmentType;
  amount: number;
  createdAt: Date;
}

```
Implicit object shapes are prohibited.  
   
⸻  
   
## ENUMS  
Business values shall use enums instead of magic strings.  
Example:  
```
enum ShipmentStatus {
  Pending,
  InTransit,
  Delivered,
  Returned,
  Refused
}

```
Magic strings inside business logic are prohibited.  
   
⸻  
   
## CONSTANTS  
Business constants shall be centralized.  
Example:  
```
/constants

shipment.ts

financial.ts

inventory.ts

marketing.ts

```
Constants shall never be duplicated.  
   
⸻  
   
## ERROR HANDLING  
Every recoverable error shall:  
* Use custom error types.  
* Preserve context.  
* Remain traceable.  
* Include business meaning.  
Generic errors are discouraged.  
   
⸻  
   
## ASYNC CODE  
Use:  
async / await  
Avoid:  
Nested Promise chains.  
Business workflows should remain readable.  
   
⸻  
   
## NULL HANDLING  
Null values shall be handled explicitly.  
Preferred:  
```
if (!settlement) {
   throw new SettlementNotFoundError();
}

```
Never assume data exists.  
   
⸻  
   
## OPTIONAL VALUES  
Optional values shall be validated before use.  
Unsafe optional chaining inside business calculations is discouraged.  
Business correctness has priority.  
   
⸻  
   
## IMPORT ORDER  
Recommended import order:  
1. External Libraries  
2. Internal Modules  
3. Shared Types  
4. Utilities  
5. Styles  
Imports remain organized.  
   
⸻  
   
## EXPORTS  
Prefer:  
Named Exports  
Avoid:  
Large default exports.  
Named exports improve discoverability.  
   
⸻  
   
## DEPENDENCY DIRECTION  
Allowed dependency flow:  
Infrastructure  
↓  
Repositories  
↓  
Services  
↓  
Business Engines  
↓  
Analytics  
↓  
UI  
Reverse dependencies are prohibited.  
   
⸻  
   
## IMMUTABILITY  
Prefer immutable data structures.  
Avoid mutating objects during business calculations.  
Business functions should return new values whenever practical.  
   
⸻  
   
## PURE FUNCTIONS  
Business formulas should be pure.  
Inputs  
↓  
Calculation  
↓  
Output  
No hidden side effects.  
   
⸻  
   
## LOGGING  
Business logging shall include:  
* Business Context  
* Entity ID  
* Formula Version  
* User (when applicable)  
* Timestamp  
Sensitive information shall never be logged.  
   
⸻  
   
## CONFIGURATION  
Hardcoded configuration is prohibited.  
Use centralized configuration files.  
Business configuration belongs outside business logic.  
   
⸻  
   
## TESTABILITY  
Every service shall support dependency injection where practical.  
Business modules should remain independently testable.  
   
⸻

## REPOSITORY PATTERN  
Repositories are responsible only for:  
* Reading Data  
* Writing Data  
* Querying Data  
Repositories shall never:  
* Calculate KPIs  
* Apply Business Rules  
* Execute Financial Logic  
* Perform FIFO Calculations  
Business logic belongs to Business Engines.  
   
⸻  
   
## SERVICE PATTERN  
Services coordinate business workflows.  
Example:  
Validate Input  
↓  
Execute Business Rule  
↓  
Persist Data  
↓  
Generate Audit Record  
↓  
Return Result  
Services remain deterministic.  
   
⸻  
   
## VALIDATION PATTERN  
Every business operation shall follow:  
Input Validation  
↓  
Business Validation  
↓  
Authorization Validation  
↓  
Execution  
↓  
Audit  
↓  
Return  
Validation is never skipped.  
   
⸻  
   
## API RESPONSE STANDARD  
Recommended response format:  
```
{
  success: true,
  data: ...,
  metadata: ...,
  timestamp: ...
}

```
Errors:  
```
{
  success: false,
  error: {
      code,
      message,
      details
  }
}

```
API responses remain predictable.  
   
⸻  
   
## DOCUMENTATION  
Every exported function shall include documentation.  
Recommended format:  
Purpose  
Inputs  
Outputs  
Business Notes  
Repository Reference  
Documentation explains business meaning.  
   
⸻  
   
## DEPRECATION POLICY  
Deprecated code shall:  
* Be documented.  
* Include migration guidance.  
* Remain temporary.  
* Be removed in scheduled releases.  
Deprecated code shall never become permanent.  
   
⸻  
   
## CODE REVIEW CHECKLIST  
Before accepting code verify:  
✓ Repository Structure respected.  
✓ Type Safety maintained.  
✓ Business Rules respected.  
✓ No duplicated formulas.  
✓ No duplicated business logic.  
✓ Tests updated.  
✓ Documentation updated.  
✓ Performance acceptable.  
✓ Security requirements satisfied.  
Code review is mandatory before merge.  
   
⸻  
   
## GIT COMMIT STANDARD  
Recommended commit format:  
```
feat(financial): add shipping subsidy engine

fix(inventory): correct FIFO layer restoration

refactor(formula): simplify dependency graph

docs(repository): update project bible

test(financial): add revenue recognition tests

```
Commits should communicate business intent.  
   
⸻  
   
## PULL REQUEST STANDARD  
Every Pull Request shall include:  
* Summary  
* Business Reason  
* Related Repository Documents  
* Test Results  
* Screenshots (UI)  
* Migration Notes (if applicable)  
Large undocumented Pull Requests are prohibited.  
   
⸻  
   
## CLAUDE CODE REQUIREMENTS  
Every implementation generated by Claude shall be:  
* Strongly Typed  
* Modular  
* Self-Documented  
* Production Ready  
* Repository Compliant  
Generated code should require minimal manual cleanup.  
   
⸻  
   
## SUCCESS CRITERIA  
The Coding Standards are considered complete only if:  
* Every file has one responsibility.  
* Every business concept has descriptive names.  
* Business logic remains centralized.  
* Code remains strongly typed.  
* Repository structure is respected.  
* Validation is consistent.  
* Documentation remains synchronized.  
* Tests remain maintainable.  
* Code reviews follow documented standards.  
* Claude generates predictable production-quality code.  
Coding standards protect the long-term maintainability of the platform.  
Consistency is a feature, not an afterthought.
