**PURPOSE**  
  
This document defines the official database architecture for the platform.  
  
It specifies the logical database design.  
  
It does NOT define implementation details for a specific database engine.  
  
Whether PostgreSQL, Supabase PostgreSQL, MySQL or another relational database is used, the logical model defined here must remain unchanged.  
  
⸻  
  
**DATABASE PHILOSOPHY**  
  
The database is not merely a storage system.  
  
It is the permanent business memory of the platform.  
  
The database must prioritize:  
  
* Financial Accuracy  
* Historical Integrity  
* Auditability  
* Scalability  
* Performance  
* Maintainability  
  
The database must never prioritize convenience over correctness.  
  
⸻  
  
**DATABASE DESIGN PRINCIPLES**  
  
The database shall follow these principles:  
  
* Normalized business entities.  
* Immutable business history.  
* Referential integrity.  
* Explicit relationships.  
* No duplicated business logic.  
* No calculated financial values permanently stored unless explicitly documented.  
* Soft deletion whenever business history must be preserved.  
* Strict foreign key relationships.  
  
⸻  
  
**DATABASE ARCHITECTURE**  
  
The database is divided into logical domains.  
  
Each domain owns its own entities.  
  
Domains include:  
  
* Core  
* Products  
* Orders  
* Shipping  
* Inventory  
* Finance  
* Marketing  
* Analytics  
* AI  
* Audit  
* Settings  
* System  
  
Domains communicate through foreign keys.  
  
Business logic never exists inside database relationships.
